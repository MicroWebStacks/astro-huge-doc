## 1. Approach: OKF-native by default, best-effort matching

Two strategies were on the table:

1. **Opt-in detection** ‚Äî identify OKF-compliant content and switch on an "OKF mode" that reinterprets it.
2. **OKF-native defaults** ‚Äî make the engine's default behavior coincide with OKF semantics wherever they don't conflict with existing behavior.

**Decision: OKF-native (option 2), refined by the rulings to "best-effort OKF matching".** We are not strictly-OKF-compliant and not OKF-only: the engine takes everything useful OKF offers (path identity, reserved files, relations from links, recommended metadata fields) as native default behavior, per-field and per-file ‚Äî never as a global mode switch. There is **no `okf.enabled` flag, no compliance detection, no compliance reporting**: if `type` is present the page gets concept treatment, if not it gets generic treatment, and nothing warns about non-compliance (OP-2, OP-5). Diverging from the v0.1 draft where our design is better is explicitly acceptable (RK-3).

## 5. Rendering step (second phase focus)

Reading-first, per handoff ¬ß13‚Äì14 ‚Äî the document stays the center, semantics live at the edges:

1. **Frontmatter viewer / concept header** (TP-9) ‚Äî every page with frontmatter gets the expandable viewer; typed pages get the concept header on top of it. Identity (concept id = slugified path) inspectable there.
2. **Breadcrumb + relations footer** (TP-10) ‚Äî this is where the historic "link neighboring pages" ideas get enforced rather than revisited: breadcrumb on top for orientation, and at the bottom prev/next plus contextual backlinks ("Referenced under *Calculation*: '‚Ä¶MRR is calculated from‚Ä¶'"). Context comes free from the relations table (link text + source heading).
3. **`log.md` special view** (TP-2/DD-8) ‚Äî when validation passes, an icon next to the info action opens the curated history in a dedicated view.
4. **Graph** (TP-13) ‚Äî optional, neighborhood-first, never the primary interface. A per-concept neighbors JSON keeps it static-build compatible and avoids loading any global graph (handoff ¬ß26). Implementation ruled in DD-11 (adjacency maps, radial SVG in panzoom, RelationsFooter entry). Full-graph explorer only if the neighborhood view proves insufficient.

## 6. Suggested delivery stages

1. **Stage 1 ‚Äî Native identity & recognition** (TP-1..4, TP-15): slugified path identity, landing priority list, schema columns, frontmatter salvage. *Engine becomes OKF-native; URLs change for title-slug sites (clean break, RK-1).*
2. **Stage 2 ‚Äî Relations & reading UI** (TP-5..10, TP-14): resolution pass, relations table per backend, backlinks, link rendering states, frontmatter viewer/concept header, breadcrumb + relations footer. *The visible release.*
3. **Stage 3 ‚Äî Navigation & exploration** (TP-11, TP-12, TP-16): authored indexes, type/tag pages + `/explore` entry points, data-inconsistency report, lite background link indexer with progress UI.
4. **Stage 4 ‚Äî Graph & beyond** (TP-13 per DD-11, multi-bundle, log timelines deep features, context export). *The former "background relations crawler for lite" item is dissolved into Stage 3's TP-16 (DD-13).*

Stages 1+2 together correspond to the handoff's "minimum useful OKF support" (¬ß31) minus type/tag explore pages (Stage 3), and deliberately dropping its conformance-status ideas per OP-5.

## 7. Rulings log

All DD/OP/RK items were ruled by the user on 2026-07-19; resolutions are recorded in the table in ¬ß3 (Resolution column) and propagated into ¬ß4‚Äì¬ß6. Notable deltas from the original proposal:

- No legacy identity mode and no redirect map ‚Äî clean URL break (RK-1).

- Relations storage is backend-specific with a capped, lossy, incrementally-grown in-memory store in lite (DD-5, RK-2).
- `log.md` gets a validation gate before receiving its special view (DD-8).
